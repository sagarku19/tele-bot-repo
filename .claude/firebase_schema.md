Single Firestore database, accessed from both projects via `firebase-admin` with service-account credentials. Three collections.

## `users` — document ID = stringified Telegram user ID
| Field | Type | Notes |
|---|---|---|
| `telegramId` | number | Numeric Telegram user ID. **Document ID is the string form.** Bot writes both. |
| `name` | string | First+Last from Telegram, or whatever the user typed first if onboarding overwrote it. |
| `username` | string | Telegram `@username` (no `@`) or empty. |
| `stage` | string | `'new' \| 'engaged' \| 'interested' \| 'payment_pending' \| 'paid'`. See [[bot-funnel-stages]]. |
| `isPaid` | boolean | True once any course is granted. |
| `paidCourseIds` | string[] | Course IDs the user owns. Appended in `flows/access.js`. |
| `selectedCourseId` | string | (Optional) Set when transitioning to `payment_pending`. Used by `handlers/photo.js` to know which course the screenshot is for. |
| `createdAt` | ISO string | Set on first `/start`. |
| `lastSeen` | ISO string | Bumped by every `updateUser`/`updateStage` call. |

## `courses` — document ID is the course `id` (slug from seed config)
| Field | Type | Notes |
|---|---|---|
| `id` | string | Slug, e.g. `prelims-2026`. Same as doc ID for seeded courses; auto-generated by Firestore for courses created from the admin panel (admin POST does **not** set `id`, so admin-created courses have a random doc ID and no `id` field — minor inconsistency, see [[admin-known-issues]]). |
| `name` | string | Display title. |
| `description` | string | Short pitch. |
| `price` | number | INR rupees (integer). |
| `channelId` | string | `@handle` or numeric `-100…` chat ID. Bot must be admin of this chat to create invite links. |
| `groupId` | string | Same as channelId. |
| `welcomeMessage` | string | Sent on access grant. |
| `active` | boolean | Used as soft-delete flag by `/api/courses DELETE`. Bot does NOT filter on this — seeded courses have no `active` field, only admin-created ones do. |
| `createdAt` | ISO string | Set on seed/create. |

**Seed data:** `upsc-bot/config/courses.config.js` ships 3 courses: `prelims-2026` (₹4999), `mains-answer-writing` (₹6999), `current-affairs-monthly` (₹999). Seeded idempotently on bot boot.

## `payments` — document ID is auto-generated by Firestore
| Field | Type | Notes |
|---|---|---|
| `telegramId` | string | **Stored as string** (`String(telegramId)`). Users doc ID is also string, so they match. |
| `courseId` | string | The selected course slug. |
| `screenshotUrl` | string | A `https://api.telegram.org/file/bot<TOKEN>/...` URL — **contains the bot token**. See [[security-considerations]]. |
| `status` | string | `'pending' \| 'verified' \| 'rejected'`. |
| `geminiAnalysis` | string \| null | **Stored as a JSON-stringified blob** of the verification result object (`{ isValid, amount, confidence, isGiftCard, notes, ... }`), not as a nested object. The admin's PaymentCard renders it as raw text — readable but ugly. |
| `verifiedAt` | ISO string \| null | Set only when status flips to `'verified'`. |
| `createdAt` | ISO string | Set on save. |

**Why:** keeping `geminiAnalysis` as a string was an early-stage decision to avoid Firestore-side schema bumps if the verification fields change. Don't change it without updating the admin's `PaymentCard.js` render path.

**How to apply:** when querying or writing, match the type. `users` doc IDs and `payments.telegramId` are strings; `users.telegramId` field is a number. Bugs hide here.
